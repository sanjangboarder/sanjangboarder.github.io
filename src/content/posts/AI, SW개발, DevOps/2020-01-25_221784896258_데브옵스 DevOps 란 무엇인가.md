**DevOps의 개념**

DevOps 라는 말은 Deveopment , Operations 의 합성어로 개발과 운영이 동시에 이루어지는 의미라고 보시면 됩니다. 기존의 개발과 운영은 개발팀에서 필요한 사항(서버, 컴파일러, 소스 저장소 등)을 인프라 부서(운영팀)에 요청하면 서버를 구매하고 개발환경을 갖추어 개발팀에 제공을 해주기 떄문에 절차도 복잡하고 적기에 필요한 환경을 구성하기에 어려운 점이 많았습니다. 이에 기존의 개발과 운영간의 벽으로 인한 비효율적인 방식을 해결해 보고자 하는 내용입니다.

​

DevOps는 Dev(개발)과 Ops(운영)이 긴밀하게 협업, 통합하여 개발생산성/효율성을 높이려는 문화/철학이라고 이해하시면 될 것 같습니다. DevOps는 최신의 기술 트랜드와 툴을 사용하는것 뿐만 아니라 개발문화, 조직구성을 하는 방식까지 아우르는 개념이며 궁극적으로는 지속적인 개선이 진행되는 시스템의 운영까지 넓은 범위를 포함한다고 볼 수 있습니다. 이를 통해 민첩하고 지속적인 개발문화를 만들고 다양한 자동화를 통해 개발 및 운영팀의 효율을 높이고, 더 빠르게 혁신하여 고객에게 더 높은 가치를 제공하기 위한 활동입니다.

**DevOps의 목적**

DevOps의 목적은 SW개발과 IT팀(운영) 간의 프로세스를 자동화하여 소프트웨어를 보다 빠르고 안정적으로 빌드, 테스트, 릴리즈 할 수 있도록 하는 것입니다. 이를 달성하기 위해서는 팀간의 협업문화 구축, SW 릴리즈 가속화, 이슈에 대한 빠른 대응, 비계획된 작업을 효율적으로 처리하는 내용이 필요합니다.

​

> DevOps의 목적 :
>
> 개발 전체적으로 배포, 낮은 에러율, 릴리즈관리 등을 DevOps 접근법으로 점차 동적 프로그래밍 시켜서, 운영 프로세스의 예측성, 효율성, 보안성, 관리성을 최대화 하는것이다.
>
> 위키피디아

**DevOps 전개 방안**

사실 SW개발하는 회사라고 하면 보통 개발팀과 운영팀(인프라)는 별도의 조직으로 존재하는 경우가 많다. DevOps에서는 이 벽을 허물기 위한 노력이 많이 필요한데, 실제 현업에서는 DevOps의 전개는 운영팀(인프라)를 담당하는 조직에서 좀 더 주도적인 액션이 필요하고 개발팀은 개발환경/인프라에 대한 요청을 상세하고 해주고 결과에 대한 피드백을 잘 해주는것이 중요하게 된다.

필자는 임베디드 제품을 만들고 있는 조직에서 일하고 있어서 DevOps 를 전개하라고 한다면 아래의 순서로 우선순위를 두어서 준비를 하게 될것이다. 순서는 있지만 Waterfall 처럼 윗단계가 진행되어야 아랫단계가 되는것은 아니다 단지 우선순위라고 이해해주기 바란다.

​

1. Public Cloud / Containers

2. Version Control

3. Tools

4. Infrastructure as code

5. Agile Planing / Lean Project Management

6. Continous Integration

7. Continous Delivery

8. Monitoring and Logging

​

이번절 역시 많은 내용이 있기 떄문에 추후 DevOps 전개시의 고려해야 할 점에 대해 별도의 포스팅을 해볼 생각이다.

​

**DevOps 의 Keyword**

데브옵스(DevOps)를 나타내는 키워드를은 상당히 많습니다. DevOps를 도입한다, 우리 조직은 DevOps를 잘 하고 있다를 볼때 아래 키워드들이 어떻게 전개되고 실제 운영되는지 살펴보면 DevOps를 잘 하고 있다고 볼 수 있습니다.

- 개발과 운영조직의 소통, 조화, 상호작용

- 개발과 운영이 하나로 협업, 통합된 관리

- 소통, 협업이 강조되는 개발환경과 문화

- 개발 요구사항에 대한 즉각적인 대처

- 지속적인 피드백과 딜리버리와 배포, 운영, 통합이 유지되는 사이클
