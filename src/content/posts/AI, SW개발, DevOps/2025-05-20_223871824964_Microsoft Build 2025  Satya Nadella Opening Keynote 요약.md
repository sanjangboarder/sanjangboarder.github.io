안녕하세요 산장보더입니다.

오늘 25년 5월20일 마이크로소프트에서 MS Build 라는 행사가 있었는데요. AI 로 앞서가는 기업답게 예상가능 하지만 놀라운 새로운 것들을 많이 발표했습니다. 저도 AI에 관심을 가지고 많이 들여다 보고 공부하는 터라 내용을 요약해 본김에 여러분들께도 공유드려보겠습니다.

약 2시간 분량의 내용을 Gemini, Grok 등의 도움을 받아서 정리했고 제가 쭉 읽어보면서 이상하거나 잘못된 표현들은 수정을 하면서 작성했습니다.

---

개발자가 되는 건 어떤 느낌일까요? 🚀

개발자가 된다는 건 정말 마법 같은 일이에요! 어릴 적 컴퓨터에 푹 빠져 코드를 짜면서 나만의 비디오 게임을 만들던 기억이 생생하네요. 그때부터 개발이 주는 즐거움에 푹 빠졌고, 개발자가 되면서 세상이 훨씬 넓어지는 기분이었어요.

특히 AI를 생각하면 가슴이 두근거려요. 처음 개발을 시작했던 이유가 바로 이 놀라움 때문이었거든요. 이렇게 강력한 도구로 우리가 무엇을 만들 수 있을지 정말 기대돼요. AI와 함께 프로그래밍하는 건 마치 제 몸의 일부가 확장되는 것처럼 든든하고, 더 멀리 나아갈 수 있도록 도와준답니다.

![](https://mblogthumb-phinf.pstatic.net/MjAyNTA1MjBfMjc4/MDAxNzQ3NzIzNjAzNjUz.vFPrBQpGo4DDypl8ijSDPg3zR_G6D_QH7jlpOm3kySsg.0x0Xc5AUKYoyNGxZewf30tgzPrZPo1IAKXyhhvGETq8g.JPEG/1000000158.jpg?type=w800)

어떤 프로젝트든, 어떤 아이디어든 '너무 크다'고 느껴지지 않아요. 사람들이 제가 만든 프로그램을 사용해서 더 나은 삶을 만들고, 세상에 좋은 영향을 주는 것을 보는 것이 가장 큰 행복이에요. 제 아이디어는 정말 무한하답니다. 우리가 하는 모든 일은 다른 사람들이 세상을 더 좋게 만들도록 돕는 것이니까요!

마이크로소프트 빌드 2025, 왜 중요할까요? ✨

안녕하세요! 빌드에 오신 것을 환영합니다! 특히 지금처럼 중요한 시기에 빌드에 다시 오게 되어 늘 즐거워요. 우리는 지금 또 다른 플랫폼 변화의 중간 단계에 진입하고 있어요. 바로 이 중간 단계에서 모든 혁신이 일어나고, 모든 것이 확장되죠.

예전에 **Win32(91년), 웹 스택(96년), 클라우드와 모바일(2008년)**이 그랬던 것처럼요. 그리고 2025년인 지금, 우리는 **'개방적인 에이전트 기반 웹(open agentic web)'**을 대규모로 구축하고 있어요. 몇 가지 앱으로만 구성된 수직적으로 통합된 방식에서 벗어나, 개방적이고 확장 가능한 에이전트 기반 웹을 가능하게 하는 플랫폼으로 나아가고 있답니다.

![](https://mblogthumb-phinf.pstatic.net/MjAyNTA1MjBfNTMg/MDAxNzQ3NzIzNjEzODM4.5RJLIWMLqPb-e0sJJtKL1yR6zsYUpKrrdOFWCKjYEh0g.HTrRsnpPxxje5W0ZRVATUmpoT8MGWo-Z6CupGKJn204g.JPEG/1000000159.jpg?type=w800)

무엇보다 중요한 건, 개발자들이 스택의 모든 계층에서 기회를 넓히는 것이에요. 여러분 모두가 지구상의 모든 사람과 모든 조직에 힘을 실어줄 수 있는 앱과 에이전트를 만들 수 있도록 말이죠. 이번 컨퍼런스에서 이 모든 것을 자세히 살펴볼 거예요. 하지만 모든 것이 시작되는 곳, 즉 우리가 구축하는 데 사용하는 도구부터 이야기해 봅시다!

개발 도구, 얼마나 좋아졌을까요? 🛠️

소프트웨어 엔지니어링은 언제나 아이디어를 현실로 만들고, 복잡성을 다듬어 완벽하게 제어하기 위해 올바른 도구를 갖는 것이 중요했어요. 우리는 이 도구들을 계속해서 발전시키고 있답니다. 정말 엄청난 성장과 확산을 보고 있어요.

실제로 Visual Studio와 그 관련 도구들은 이제 5천만 명이 넘는 사용자를 보유하고 있고, GitHub는 1억 5천만 명의 사용자가 사용하고 있죠. 그리고 GitHub Copilot은 무려 1,500만 명 이상의 개발자가 사용하고 있답니다. 이건 정말 시작에 불과해요!

![](https://mblogthumb-phinf.pstatic.net/MjAyNTA1MjBfMTk2/MDAxNzQ3NzIzNjIzNzAz.peK-ZlniTFYMiDuiF7DR3sAo6CNWF7J29LPvhamxxswg.rNE1wb5rJ54RXj0nooqpc8l9EzB1MkNArbt13ZTT4O4g.JPEG/1000000160.jpg?type=w800)

이번 빌드에서 Visual Studio를 시작으로 많은 새로운 업데이트를 출시하고 있어요. Visual Studio는 .NET과 C++를 위한 가장 강력한 IDE이며, 우리는 이것을 더욱 발전시키고 있답니다. .NET 10 지원, 디자인 타임에서의 라이브 미리 보기, Git 도구 개선, 크로스 플랫폼 앱을 위한 새로운 디버거 등 정말 많은 기능이 추가되었어요.

VS Code는 불과 몇 주 전에 100번째 오픈 소스 릴리스를 출시했답니다. 개선된 멀티 윈도우 지원 기능이 포함되었고, 에디터 내에서 바로 변경 사항을 쉽게 볼 수 있도록 했죠. GitHub는 계속해서 개발자들의 든든한 보금자리가 되고 있어요. GitHub Enterprise는 기업 환경에서 엄청난 기세로 확산되고 있으며, 우리는 어떤 애플리케이션이든 구축하는 개발자들을 위해 두 배로 투자하고 있답니다. 신뢰, 보안, 규정 준수, 감사 기능, 데이터 상주성은 오늘날 더욱 중요해졌으니까요!

GitHub Copilot, 이제 단순한 코딩 도우미가 아니라고요? 🤯

이제부터 여러분이 사용하는 도구부터 배포하는 인프라까지, **'신뢰'**에 대해 이야기해 볼게요. 오픈 소스는 GitHub의 핵심이에요. 그리고 우리는 다음 단계로 나아가고 있답니다. GitHub Copilot이 VS Code 내에서 발전하면서, AI는 우리가 코딩하는 방식의 중심이 되었어요. 그래서 우리는 VS Code에서 Copilot을 오픈 소스로 전환하고 있답니다.

이건 정말 큰 변화예요! 오늘부터 우리는 AI 기반 기능을 VS Code의 핵심 기능으로 직접 통합할 거예요. 전 세계에서 가장 사랑받는 개발 도구를 구동하는 동일한 오픈 소스 저장소에 포함시키는 거죠.

그리고 물론, 우리는 계속해서 GitHub Copilot 2를 구축할 거예요. 지난 몇 년간 코드 자동 완성, 채팅, 멀티 파일 편집을 거쳐 이제는 에이전트까지 발전했죠. 이 동일한 패턴이 에이전트 기반 웹 전반에 걸쳐 나타나고 있어요.

질문을 하면 AI 어시스턴트가 답변을 해주고, 에이전트에게 작업을 할당하고 실행하게 하거나, AI와 함께 작업해서 프로젝트를 완료할 수도 있죠. 이 모든 형태를 혼합해서 사용할 수도 있답니다. 이건 개발자로서 우리가 중요하게 생각하는 점이에요. 어떤 하나만 중요한 게 아니거든요.

![](https://mblogthumb-phinf.pstatic.net/MjAyNTA1MjBfNTEg/MDAxNzQ3NzIzNjMxNzc4.YxSax9fJ8mlxAo6ovAopW-LwgdY7Tbxk8WRNpxoclhIg.GlfWh6FceRKKkHaqw4mPyL1csGuNtxzKgjSn3Zrbcz8g.JPEG/1000000161.jpg?type=w800)

우리는 앱 현대화 기능을 에이전트 모드에 바로 구축하고 있어요. 이제 Copilot은 Java 8을 Java 21로 업그레이드하거나 .NET 6을 .NET 9으로 업그레이드하는 등의 프레임워크 업그레이드를 할 수 있어요. 또한 온프레미스 앱을 클라우드로 마이그레이션할 수도 있죠. 코드와 종속성에 대한 계획을 세우고, 진행하면서 해결 방법을 제안하며, 여러분이 변경한 내용을 학습해서 전체 프로세스를 원활하게 만들어요.

다음으로 소개하는 것은 SRE(Site Reliability Engineering)를 위한 자율 에이전트예요. 생각해 보세요. 개발자에게 가장 큰 어려움 중 하나가 한밤중에 깨어나 라이브 사이트 문제를 해결하는 것이죠. SRE 에이전트가 자동으로 문제를 분류하고, 근본 원인을 파악하고, 완화 조치를 취해요. 그리고 모든 복구 항목이 포함된 사고 관리 보고서를 GitHub 이슈로 기록하죠. 거기서부터 복구 항목을 GitHub Copilot에 할당할 수도 있답니다.

우리는 여기서 멈추지 않아요. GitHub에 완전히 통합된 코딩 에이전트, 즉 Copilot을 페어 프로그래머에서 피어 프로그래머로 만드는 것이 다음 큰 발전 단계예요. 여러분은 버그 수정, 새로운 기능 추가, 코드 유지 보수와 같은 이슈를 Copilot에 할당할 수 있고, Copilot은 이러한 작업을 자율적으로 완료할 거예요. 오늘, 이 기능이 여러분 모두에게 제공된다는 것이 정말 기뻐요!

AI 에이전트, 정말 똑똑하게 일할 수 있을까요? 🧠

이제는 단순히 버그를 보고하는 시대를 넘어섰어요. 이제는 버그를 수정하도록 할당할 수 있죠. 이게 바로 **'권한 부여(empowerment)'**예요. 제가 GitHub 이슈에서 다루어야 할 모든 버그나 이슈들이 여기에 있어요. 첫 번째는 사용자 그룹 크기 커뮤니티 페이지에 필터를 추가하는 것이에요. 이 이슈를 살펴보죠. 여기 새로운 필터를 추가해야 한다고 나와 있어요. 크기별로 소형, 중형, 대형 범위를 지정해야 하는 것 같아요. 어떤 종류의 백분위수 같은 것이죠. 그룹별 케이스 구문 같은 것이에요.

가장 쉬운 방법은 제 새로운 친구인 Copilot에게 할당하는 거예요. 할당하고 나면 Copilot이 그 문제를 받아들였고, 저를 인식하고 작업을 시작하는 것을 볼 수 있어요. 나중에 다시 확인해 볼 거예요. 이렇게 이슈를 처리하는 것이 정말 즐거워요. 이메일 분류를 Copilot에게 할당하는 것과 같죠. Copilot은 브랜치를 설정하고, GitHub Actions를 시작해요. 즉, 계산 환경을 생성하거나 GitHub Actions를 사용해서 가상 머신을 만들어요. 세션 로그에 초안 PR(Pull Request)을 커밋하고, 작업하는 동안 세션 로그로 돌아가서 모든 초안 PR을 계속 확인할 수 있죠.

우리의 코딩 에이전트는 훌륭한 개발자 경험을 제공하는 동시에 모든 보안 조치를 준수해요. 에이전트는 자체 브랜치에서 작업하며, 개발자가 구성한 MCP 서버만 사용하죠. 다른 에이전트에게 코드 리뷰를 요청하고, CI/CD를 실행하거나 병합하기 전에 사람들에게 상황을 알릴 수도 있어요. 우리는 또한 이러한 모든 기능들을 파트너들이 활용할 수 있도록 해서, SRE, SWE, 코드 리뷰 등 여러분 모두가 구축할 다양한 에이전트들의 개방적이고 안전한 생태계를 보장하고 있답니다. 물론, 개인 개발자뿐만 아니라 IT 부서도 모든 제어권을 가질 수 있도록 보장하고 싶어요.

에이전트 생태계에 대해 이야기하자면, 우리는 지난 금요일에 출시된 OpenAI의 CodeX 에이전트에 대해 매우 기대하고 있어요. 샘 알트만(Sam Altman)이 온라인으로 함께하게 되어 정말 기뻐요. 샘, 빌드에 오신 것을 환영해요!

OpenAI와 마이크로소프트의 협력, 어떤 의미일까요? 🤝

개발자들이 소프트웨어 엔지니어링을 위해 사용하는 다양한 형태에 대해 샘도 많이 생각했을 거예요. CLI를 개발했고, 지난주에는 코딩 에이전트를 출시했죠. 샘, 소프트웨어 엔지니어링이 어떻게 발전하고 개발자들이 이 다양한 형태들을 어떻게 함께 사용할지에 대한 비전에 대해 이야기해 줄 수 있을까요?

Microsoft 365 Copilot, 우리의 업무를 어떻게 바꿔줄까요? 💼

저는 항상 이것이 AI를 위한 UI라고 말해요. 예를 들어, 채팅은 웹 데이터와 여러분의 업무 데이터 모두에 기반하고 있어요. 특히 페이지와 함께라면 게임 체인저죠. 검색은 Microsoft 365 데이터뿐만 아니라 Confluence, Google Drive, Jira, ServiceNow와 같은 모든 애플리케이션에서 작동해요.

노트북을 사용하면 이제 다양한 종류의 데이터를 모아서 관리할 수 있어요. 채팅, 페이지, 모든 문서, 이메일 등을 하나의 컬렉션에 담을 수 있고, 심지어 오디오 리뷰나 팟캐스트도 추출할 수 있죠. 파워포인트를 새로운 설명 비디오로 바꾸거나 이미지를 생성하는 데 '만들기' 기능을 사용할 수 있어요.

에이전트에 대해서는 두 가지 특별한 에이전트가 있어요. 바로 **연구원 에이전트(researcher agent)**이죠. 웹과 기업 소스를 종합해서 어떤 주제나 프로젝트에 대해서든 깊이 있는 사고 과정을 적용하기 때문에 저에게는 가장 큰 게임 체인저였어요. **분석가 에이전트(analyst agent)**는 여러 소스 파일의 원시 데이터를 처리해요. 엑셀 파일 여러 개를 업로드하기만 하면 인사이트를 얻고, 예측을 하고, 모든 시각화를 해준답니다. 이러한 에이전트들은 결국 전문가의 지식을 여러분 손끝에 가져다주는 것이에요. 빌 게이츠는 오래전에 여러분 손끝에 정보가 있다고 이야기했어요. 우리는 이제 전문 지식을 여러분 손끝에 가져다주는 시대에 와 있다고 생각해요.

![](https://mblogthumb-phinf.pstatic.net/MjAyNTA1MjBfODEg/MDAxNzQ3NzIzNjY4MTEw.ZokU3sb-iD5kO58UpRx_JQJdqiQT6bR_d1tsPB50Ck4g.29tkqAWmszuuu7H1kTQ7hGMo8mAOoYTWi11QhtoX9qUg.JPEG/1000000163.jpg?type=w800)

팀즈는 이 모든 것을 가져와서 멀티플레이어로 만들어요. 여러분이 구축한 모든 에이전트가 이제 팀즈와 코파일럿에 나타날 수 있죠. 채팅이나 회의에서 에이전트를 @멘션하는 것만으로도 질문을 하고, 실행 항목을 할당하거나, 워크플로우를 시작할 수 있어요. 팀즈 AI 라이브러리를 사용하면 멀티플레이어 에이전트를 구축하는 것이 그 어느 때보다 쉬워졌어요. 이제 MCP를 지원하며, 한 줄의 코드만으로 A2A를 활성화할 수도 있어요. Azure Search와 새로운 검색 시스템을 사용해서 에피소드 메모리 또는 의미론적 메모리와 같은 기능도 추가할 수 있답니다.

개발자로서 이제 에이전트를 구축하고 게시할 수 있어요. 이것이 가장 큰 장점이죠. 에이전트 스토어에 에이전트를 게시해서 코파일럿과 팀즈에서 검색되고 배포되도록 할 수 있으며, 수억 명의 사용자에게 접근하고 기회를 얻을 수 있죠.

지난 한 해 동안, 우리는 코파일럿과 팀즈에 연결되는 다양한 산업 분야의 수많은 파트너들이 에이전트를 구축하는 것을 보았어요. 몇 가지 예시를 공유해 볼게요.

\* Workday의 에이전트를 사용하면 코파일럿에게 "주의해야 할 사항이 무엇인가요?"라고 물을 수 있어요. 그러면 Workday 작업에 대한 요약을 제공하여 학습, 승인, 워크플로우를 놓치지 않게 도와주죠.

\* ServiceNow 에이전트를 사용하면 실시간으로 사건에 대해 질문하고, 모든 해결 메트릭스를 파악한 다음, 이를 사용하여 모든 결과가 포함된 파워포인트 프레젠테이션을 즉시 만들 수 있어요.

\* LSEG의 에이전트를 사용하면 금융 전문가가 거래소에서 제공하는 모든 금융 데이터를 엑셀과 파워포인트에서 바로 검색, 분석, 공유할 수 있답니다.

Copilot Studio를 사용하면 나만의 에이전트를 구축하기 시작할 수 있어요.

나만의 AI 에이전트를 만들 수 있다고요? Copilot Studio! 🎨

Copilot Studio에는 많은 새로운 기능들이 추가되었어요. 최근에는 전체 코어 에이전트, MCP 에이전트 흐름을 Copilot Studio에 바로 넣었죠. 그리고 이것이 매우 중요해요. 이제 LLM과 결정론적 워크플로우를 모두 혼합해서 사용할 수 있답니다. 오늘 우리는 오케스트레이션을 사용해서 Copilot Studio에서 훨씬 더 복잡한 다중 에이전트 워크플로우를 더 쉽게 구축할 수 있도록 하고 있어요.

신입 사원 온보딩과 같은 작업을 생각해 보세요. 사실 꽤 복잡한 프로세스예요. 시설, 재무, 법률 분야의 에이전트들이 각자의 전문 지식과 워크플로우를 가지고 참여하죠. 이 모든 에이전트들을 함께 모으면 모든 것이 더 빠르게 진행되고, 모든 사람에게 더 나은 경험을 제공할 수 있어요. 바로 이것이 **Copilot Studio의 다중 에이전트 오케스트레이션(multi-agent orchestration)**이에요.

![](https://mblogthumb-phinf.pstatic.net/MjAyNTA1MjBfMjg1/MDAxNzQ3NzIzNjgyNzA3.a2GPnIxU-QUI6nxRBMI_sMztWPtHb2CzPOCAa0TOpJEg.c3x-p0iUeNwlUfkiwQrk_ezUmw9g92wH5DQsvJ6rM7Qg.JPEG/1000000164.jpg?type=w800)

지난 한 해 동안 여러분은 Copilot과 팀즈에 연결되는 100만 개 이상의 에이전트를 구축했으며, 우리는 속도를 늦추지 않을 거예요.

오늘 우리는 회사의 데이터, 워크플로우, 스타일에 맞게 미세 조정된 모델을 사용하여 구축할 수 있는 새로운 종류의 엔터프라이즈급 에이전트를 소개해요. 이것을 Copilot Tuning이라고 부르죠. 이건 정말 대단한 일이에요. 어느 정도는 단순히 Copilot을 사용하는 것이 아니라, 모든 고객, 모든 기업, 모든 회사를 위해 Copilot을 튜닝하는 것에 관한 것이죠.

이제 Copilot은 여러분 회사의 독특한 어조와 언어를 학습할 수 있으며, 곧 회사의 특정 전문 지식과 지식까지 이해하게 될 거예요. 소량의 참조 자료를 학습 환경에 제공하고 학습 실행을 시작하기만 하면 된답니다. 사용자 지정된 모델은 모든 소스 제어의 권한을 상속하며, 에이전트에 통합되면 승인된 사용자에게 배포할 수 있어요.

예를 들어, 법률 회사라면 과거 논쟁과 관련 문헌을 통해 추론하고, 회사에 매우 특화된 답변을 제공하거나 문서를 생성할 수 있어요. 또는 여러 산업 분야에서 일하는 컨설팅 회사라면 이제 각 산업 분야에 맞게 모델을 튜닝하여 해당 산업 분야의 워크플로우에 대한 특정 노하우를 반영할 수 있죠. 이것은 회사가 가진 전문 지식을 더욱 증폭시켜 회사 내 모든 사람이 그것을 이해하고, 여러분의 제품과 서비스에도 반영되도록 하는 것에 관한 것이에요.

제가 이야기한 모든 것을 좀 더 잘 느끼게 해드리기 위해, 무대에 미디를 초대해서 Copilot 생태계를 모두 보여드리도록 할게요. 미디, 부탁해요.

Azure AI Foundry, AI 개발의 핵심 플랫폼일까요? 🏗️

우리가 하고 있는 일은 Copilot, Copilot Studio 아래에 있는 모든 것을 가져와서 여러분 개발자 모두가 자체 애플리케이션, 자체 애플리케이션 확장 기능을 구축할 수 있도록 하는 최고의 플랫폼으로 제공하는 것이에요. 모델이 발전하고, 더 빨라지고, 몇 달마다 새로운 샘플이 출시되면서 더 유능해짐에 따라, 앱은 이러한 완전한 상태 유지 애플리케이션이 되어 다중 모델 및 다중 에이전트가 되어야 해요. 이것이 지금의 큰 변화이죠. 단순히 요청-응답 API 호출을 통해 하나의 모델을 사용하는 것이 아니에요.

우리는 실제 상태 유지 다중 모델 애플리케이션을 구축하고 있으며, 이들은 프로덕션 환경에서 사용할 수 있어야 하죠. 이것이 최고의 앱 서버를 구축하는 동기예요. Foundry를 지능을 위한 생산 라인처럼 생각해보세요. 이러한 에이전트와 애플리케이션을 구축하는 데는 훌륭한 모델 그 이상이 필요해요. 모델 주변의 시스템, 즉 평가, 이 오케스트레이션 레이어(orchestration layer), 또는 RAG 등 이 모든 것이 정말 중요하며, Foundry는 AI 시대를 위한 완전한 앱 플랫폼이에요.

BMW, Carvana, Coca-Cola, NASDAQ과 같은 다양한 산업 분야의 70,000개 이상의 조직이 이미 이를 사용하고 있으며, GainSight와 같은 많은 ISV들도 사용하고 있죠. 모두 Foundry를 선택하고 있으며, 기업들은 AI의 ROI를 진정으로 실현하기 위해 POC 단계에서 벗어나 전사적인 배포를 진행하고 있어요. 사실, 지난 3개월 동안 우리는 100조 개 이상의 토큰을 처리했는데, 이는 전년 대비 5배 증가한 수치예요. 최근 일본에 갔을 때 Foundry로 구축된 청각 처리 장애를 겪는 사람들이 듣는 것을 이해하도록 돕는 앱을 보았는데, 정말 놀라웠어요. 비디오를 보시죠.

Grok과 같은 최신 AI 모델도 Azure에서 만날 수 있나요? 🤖

하지만 오늘의 접근 방식은 여러분이 구축하는 앱이나 에이전트가 하나의 모델에만 연결되는 것에서 벗어나 진정으로 다중 모델(multi-modal)이 되는 것이에요. 그래서 오늘 XAI의 Grok이 Azure에 출시된다는 것을 발표하게 되어 정말 기뻐요. Grok 3는 추론, 심층 검색, 응답 모델 이 모든 것을 하나의 모델에 담고 있으며, 주말에 Elon과 이에 대해 이야기할 기회를 가졌다는 것이 정말 멋진 일이에요. 비디오를 재생해 보죠.

빌드에 함께 해주셔서 정말 감사해요, Elon. Microsoft에서 인턴으로 시작하셨고, Windows 개발자였으며, 물론 여전히 PC 게임을 좋아하신다고 들었어요. Windows와의 초기 시절과 어떤 종류의 것들을 구축하셨는지 이야기해 주시겠어요?

![](https://mblogthumb-phinf.pstatic.net/MjAyNTA1MjBfMTU2/MDAxNzQ3NzIzNjk0Mzgz.coBO1SWbTOZ5bHSR4zgn4G7gFSYlrZt54NC4INx3Bhcg.RsJpNMKjIMCT6Xes1QjBgPIs18IHdfV1d-DqZ-P-R5Ug.JPEG/1000000165.jpg?type=w800)

네, 저는 사실 Windows 이전에 DOS로 시작했어요. 초기 IBM PC 중 하나에 MS-DOS가 있었고, 처음에는 128K, 그리고 256K로 두 배가 되었는데, 그때는 엄청 많다고 느꼈어요. 그래서 네, DOS로 비디오 게임을 프로그래밍했고, 나중에는 Windows 3.1에서 했죠.

Foundry에서는 모델 선택부터 시작해요. 응답 모델, 추론 모델, 작업별 모델, 다중 모달 등 1,900개 이상의 모델을 이미 지원하고 있어요. 그 모든 모델들이 Foundry에 있답니다. 물론, 여기에는 OpenAI 파트너의 최신 모델들도 포함돼요. 올해만 해도 OpenAI의 15개 모델을 Azure에서 동시에 출시해서 새로운 모델이 출시될 때 바로 사용할 수 있도록 했어요. Sora도 다음 주에 출시돼요.

개발자로서 우리는 비용, 신뢰성, 지연 시간, 그리고 품질과 같은 여러 차원을 중요하게 생각해요. Azure OpenAI는 동급 최고예요. 높은 신뢰성, 배치 또는 스필오버와 같은 뛰어난 비용 제어 기능, 최고의 보안, 규정 준수 및 안전과 같은 엔터프라이즈 수준의 보증을 제공해요.

하지만 모델을 선택하는 것은 여전히 번거로울 수 있어요. 쿼리를 올바른 모델로 빠르게 라우팅할 수 있어야 하죠. 우리는 이것도 더 쉽게 만들고 있어요. 새로운 모델 라우터(model router)는 작업에 가장 적합한 OpenAI 모델을 자동으로 선택할 거예요. 더 이상 수동으로 모델을 선택할 필요가 없답니다.

이제 Grok이 Azure에 출시되면서, Provisioned Throughput을 한 번 프로비저닝하면 Grok을 포함한 여러 모델에서 사용할 수 있어요. 이는 모델 및 모델 프로비저닝 방식에 대한 게임 체인저예요. 이제 이것을 수행하면 EU 지역에서 모든 주권 배포로 프로비저닝할 수 있는 Mistral도 사용할 수 있어요. 전 세계 애플리케이션을 구축하는 사람들에게 점점 더 중요한 고려 사항이 되고 있죠. 점점 더 많은 지역에서 선호하는 모델이 있을 것이라고 생각하기 때문이에요.

Mistral에 대해 매우 기대하고 있어요. Llama는 최근 Mark와 함께 LamaCon에 참석했는데, 그가 말하듯이 Llama 전체 무리, 전체 Llama 떼를 Azure로 데려오고 있답니다. 우리는 오픈 소스, Black Forest Labs 등에서 이루어지고 있는 모든 발전에 대해 기대하고 있어요. 그래서 그 모든 모델들이 이 하나의 Provisioned Throughput 뒤에 있어요. 개발자로서 이 모든 모델들을 혼합해서 사용할 수 있다는 것이 흥미로운 일이죠. 많은 모델들이 동일한 API 시그니처를 가지고 있어서 모델 전환 기능도 훨씬 쉬워졌답니다.

우리는 또한 Hugging Face와의 파트너십을 확장해서 Foundry에서 11,000개 이상의 최첨단 및 오픈 소스 모델에 액세스할 수 있도록 할 거예요.

AI 에이전트가 데이터를 똑똑하게 활용하려면 무엇이 필요할까요? 💡

하지만 아시다시피, 모델은 방정식의 일부일 뿐이에요. 에이전트를 구축하는 모든 사람에게는 실시간 웹뿐만 아니라 전체 기업 지식 그래프에 대한 훌륭한 접근 권한이 필요해요. 우리는 RAG 앱을 구축하면서 더 정교한 검색 시스템이 필요하다는 것을 알게 되었어요. 단순히 벡터 검색과 일부 임베딩만으로는 충분하지 않아요. 에이전트를 위해 특별히 구축된 데이터베이스나 지식 엔진, 진정한 쿼리 엔진이 필요하죠. 그래서 복잡한 쿼리를 분해하고 병렬로 실행하며 합성된 결과를 반환할 수 있어야 해요. 본질적으로 사용자가 무엇을 원하고 데이터가 무엇을 의미하는지 이해하는 에이전트를 위한 현대적인 지식 검색 스택이라고 생각하면 돼요.

물론, 시스템의 그 부분을 갖추고 나면 다음 계층은 **오케스트레이션 레이어(orchestration layer)**예요. 이 모든 에이전트 또는 여러 에이전트가 오케스트레이션되도록 하고 싶겠죠. Foundry 에이전트 서비스는 선언적 에이전트를 구축할 수 있도록 해줘요. 사실, 포털에서 단 몇 줄의 코드만으로도 가능하죠. 복잡한 워크플로우의 경우 다중 에이전트 오케스트레이션을 지원해요. Semantic Kernel, Autogen과 같은 프레임워크와 원활하게 통합되며, 에이전트 서비스를 본질적으로 실행 환경을 위한 관리 서비스처럼 사용할 수 있어요.

![](https://mblogthumb-phinf.pstatic.net/MjAyNTA1MjBfOTQg/MDAxNzQ3NzIzNzAzNzY5.0E6oqphKA4Jn4_7ULQml5StM1--cmrjrZxZ81Worwecg._e6Mf7xZzvriFE2k22fIQhrHAHipKESLPKJShuQJY9Ig.JPEG/1000000166.jpg?type=w800)

10,000개 이상의 조직이 이미 이를 사용하고 있으며, 에이전트 서비스가 이제 일반적으로 제공된다는 것을 공유하게 되어 기쁩니다. 에이전트를 실행하고 나면 다음으로 필요한 것은 컴퓨팅 환경이에요. 서버리스 Azure Functions부터 Azure Container Apps, 그리고 AKS를 통해 필요한 극한 제어까지 모든 것을 제공하고 있어요. 에이전트 시나리오에 대해 올바른 가격 성능을 달성할 수 있도록 전체 스펙트럼의 컴퓨팅 기능을 제공하고 있으며, 예를 들어 Foundry를 Container App 또는 Functions에 연결하고 클라우드 또는 Arc가 있는 하이브리드 모드에서 AKS에 모든 오픈 소스 모델을 배포하는 것을 간단하게 만들고 있어요. 점점 더 이러한 모델이 에지에서 배포되기를 원할 것이고, Foundry는 그것을 지원할 거예요.

그리고 우리는 Foundry와 Copilot Studio 간의 간극과 연결을 좁히고 있으며, 이는 개발자로서 우리에게 매우 중요한 부분이에요. 이제 모델을 가져와서 Foundry에서 미세 조정하거나 사후 학습시킨 다음, 바로 Copilot Studio에 넣어 해당 사후 학습 모델을 사용하여 워크플로우를 자동화하거나 에이전트를 구축할 수 있죠. 이런 종류의 작업을 수행할 수 있답니다.

이것의 환상적인 예는 스탠포드 의대가 환자 기록부터 방사선학, PubMed 데이터, 임상 시험 데이터 등 다양한 시나리오에 연결하기 위해 여러 에이전트를 사용하여 오케스트레이션한 것이에요. 하지만 한 가지 목표는 암 치료 및 종양 보드 회의와 같은 중요한 작업을 AI로 오케스트레이션하는 방법을 찾는 것이었죠. 정말 대단해요. 비디오를 보시죠.

AI가 의료 분야도 혁신할 수 있을까요? 스탠포드 의대 사례! 🏥

스탠포드 의학은 암 연구의 선두에 서 있어요. 치료의 맥락에서 종양 보드 회의는 환자가 익숙하지 않은 방식으로 나타날 때 소집되는 여러 임상의들이 모이는 정말 중요한 회의예요. 약물, 모든 시술, 방사선학, 검사 결과, 환자의 병력, 그리고 의학 문헌에 대한 정보를 모아야 하죠. 종양 보드 회의를 위해 모아야 할 정보가 많을 뿐만 아니라, 그 정보가 여러 다른 곳에 흩어져 있어요. 스탠포드는 매년 약 4,000건의 종양 보드 회의를 개최하는데, 실제 증거, 문헌, 임상 시험 등을 검토해요. 이것들은 우리가 수동으로 하는 일이며, 매년 4,000번 할 수는 없어요.

**의료 에이전트 오케스트레이터(healthcare agent orchestrator)**는 이 모든 것을 처음부터 통합하여 환자 의사 결정을 더 효율적이고, 더 빠르고, 어쩌면 더 정확하게 내릴 수 있도록 돕는 방법이에요. 이것은 Azure AI Foundry를 통해 배포 가능한 에이전트 기반 AI 솔루션이에요. 우리는 자체 에이전트를 구축하고, 사용자 지정하고, 배포할 수 있었고, 첫 번째 사용 사례는 종양 보드였어요. 모든 다른 에이전트들이 정보를 보내고, 이러한 이질적인 정보 소스를 모두 통합하는 포괄적인 보고서를 임상의에게 제공할 수 있는 오케스트레이션 레이어가 있어요.

Microsoft가 에이전트에 통합한 정말 강력한 기능은 특정 임상 메모리 내에서 정보의 근거를 찾는 것이에요. 그들은 이미 Word를 사용하여 요약을 하고 있어요. 그들은 종종 파워포인트 슬라이드를 만들죠. 이것은 모든 것을 통합된 설정에서 하나의 요약으로 만들 수 있도록 해줘요. 이 에이전트들을 팀즈에 배포하여 직접 상호 작용할 수 있게 하는 데는 단 몇 줄의 코드만 필요했어요. 이 모든 시스템 간의 통합이 많을수록 기계적인 일에 들이는 노력이 줄어들고, 우리가 전문가인 실제 의사 결정에 더 많은 노력을 기울일 수 있어요.

이것은 우리가 구축할 수 있는 플랫폼으로 제공되고 있어요. 다른 사람들과 공유하기 위해 패키징할 수 있어서 지역 병원에서도 학술 의료 센터와 동일한 종류의 기능을 이용할 수 있답니다. 우리가 더 빠르게 작업하고, 더 나은 정보를 가지고 더 나은 결정을 내린다면, 그것 자체가 혁명이에요. 우리는 스탠포드 의사뿐만 아니라 전 세계 의사들을 도울 도구를 개발하고 싶어요. 이것은 혁신적일 것이며, 스탠포드가 사용한 이 의료 에이전트 오케스트레이터가 이제 Foundry에서 모든 사람에게 제공된다는 것을 공유하게 되어 정말 기쁩니다. 정말 멋진 일이에요.

Azure AI Foundry와 GitHub Copilot, 개발자의 삶을 어떻게 편하게 만들까요? 👩‍💻

이 모든 것을 현실로 만들기 위해 동료인 카데샤를 무대로 초대해서 이 모든 것이 어떻게 하나로 합쳐지는지 보여드리도록 할게요. 카데샤, 부탁해요.

카데샤가 Azure AI Foundry와 GitHub Copilot을 사용해서 개발자 생활을 훨씬 편하게 만들 수 있는 방법을 보여줄 거예요. 이것은 Azure AI Foundry를 사용하는 채팅 기반 앱인 Vibe Travel이며, 사람들이 여행 계획을 세우도록 돕는 AI 여행 에이전트를 제공해요. 약 8개월 후에 뉴질랜드 여행을 계획하고 있어서 Vibe에게 스키 탈 장소를 물어봤어요.

음, 이 계산은 좀 이상해요. 8개월 후는 2026년 1월일 거예요. 그리고 생각해 보세요. 그때는 남반구의 여름 시즌이 될 거예요. AI 에이전트를 더 스마트하게 만들고 더 근거 있는 답변을 제공하도록 해야 해요.

Foundry로 들어가서 이 앱과 다른 앱들의 모든 AI 에이전트를 확인할 수 있어요. Vibe Travel 에이전트는 GPT-4.0을 사용하고 프롬프트는 꽤 좋지만, 약간의 지식을 추가해야 해요. 참조 데이터가 있는 파일이나 Microsoft Fabric 또는 TripAdvisor와 같은 다른 서비스에 대한 연결을 제공하여 이것을 할 수 있지만, Bing 검색으로 근거를 추가하는 것만으로도 많은 개선을 이룰 수 있을 것 같아요. 그 옵션을 선택해 보죠. 정말 간단해요. 지식에 대한 접근 권한을 부여하는 것 외에도 Flat Reservation API에 대한 접근 권한도 부여하며, 개방형 API이기 때문에 바로 작동하죠.

이제 앱으로 돌아가서 쿼리를 다시 시도해 보죠. 데이터 소스나 검색 기능을 추가할 수 있으면 에이전트가 더 스마트해지고 환각 현상을 일으킬 가능성이 훨씬 줄어든답니다. 호텔 할인 정보나 날씨 예측 정보를 추가해서 더 스마트하게 만들 수도 있어요. 가능성은 무한하죠. 약 8개월 후인 2026년 1월은 여름 시즌이고, 그런 일이 훨씬 더 많이 일어날 거예요. 2026년 1월은 여름 시즌이니 스키는 불가능해요. 훨씬 나아졌고, 개선하는 것이 얼마나 쉬웠는지 보셨을 거예요.

이제 에이전트가 제 대신 실제로 비행기를 예약하도록 해봅시다. 에이전트가 수행할 수 있는 작업을 추가하면 앱이 간단한 질문-답변 기계에서 실제로 사용자를 대신하여 일을 할 수 있는 자율 에이전트로 업그레이드돼요. 해당 예약을 제 캘린더에 넣거나, 호텔 예약을 요청하거나, 심지어 내부 시스템에서 휴가 신청을 요청하는 데 액션을 사용할 수 있어요.

이 비행기들은 멋져 보여요. Foundry가 에이전트를 개선하는 것을 쉽게 만들었다는 것을 방금 보셨지만, 실제 코드에는 아직 할 일이 많아요. GitHub Copilot의 에이전트를 사용해서 이 작업을 완료할 거예요. GitHub Copilot 에이전트 모드를 열어 보죠. 새로운 기능인데, 참고로 가장 인기 있고 강력한 모델 중에서 선택할 수 있어요. Agent Mode는 GitHub Copilot을 단순히 질문에 답하거나 코드를 제안하는 것을 넘어 실제로 작업을 수행하도록 도와줘요.

여기 와서 "이 저장소에 나에게 할당된 이슈를 보여줄 수 있나요?"라고 물을 수 있어요. 전통적으로는 github.com에서 이슈를 보기 위해 브라우저를 열었겠지만, **MCP(Model Context Protocol)**를 사용하면 GitHub Copilot과 같은 도구에 필요한 컨텍스트와 기능을 제공하여 도움을 줄 수 있어요. GitHub Copilot은 우리가 GitHub 이슈에 대해 묻고 있다는 것을 알아차리고 GitHub MCP 서버를 사용하여 해당 정보를 가져오며, 몇 가지 이슈가 저에게 할당된 것 같아요. 첫 번째 이슈를 살펴보죠.

앱에서 이 초기화 버튼을 클릭하면 아무 경고 없이 채팅 세션이 즉시 지워져요. 이상적이지 않죠. 먼저 사람들에게 경고해야 해요. 앱으로 돌아가서 GitHub Copilot에게 이슈 번호 1의 세부 정보를 구현하는 데 도움을 요청하고 이슈 번호 1의 세부 정보도 가져오도록 요청할 수 있어요. 또한 매니저가 막판에 준 멋진 디자인도 있는데, 이 디자인을 프롬프트의 컨텍스트로 추가해야 해요. 보셨으면 좋겠어요. 스크린샷을 보여드릴게요. 꽤 멋지다고 생각하지 않나요? 이것을 컨텍스트 창에 추가하고 프롬프트를 보낼 거예요.

Copilot이 GitHub MCP 서버를 사용해서 이슈의 세부 정보를 가져오기 위해 특정 권한을 요청하는 것을 볼 수 있어요. 계속을 눌러 코드베이스에 변경 사항을 구현하도록 허용해 봅시다. GitHub MCP 서버가 세부 정보를 가져온 것을 볼 수 있고, 에이전트 모드의 새로운 비전 기능을 사용하면 Copilot이 제가 원하는 스케치까지 이해할 수 있다는 것을 의미해요. 그런 다음 변경하고 싶은 이슈들을 파악하고 변경하는 방법을 알아내기 시작해요.

그동안 다른 이슈를 살펴보죠. 사실, 저는 이 이슈에 대해 약간 속임수를 썼어요. 무대에 오기 전에 사티아가 Azure Dev 커뮤니티 웹사이트로 보여주었던 것처럼 이 이슈를 Copilot에게 할당했답니다. Copilot이 이 이슈를 살펴보고 우리를 위해 풀 리퀘스트를 열었어요. 살펴보죠. 여기에 구현 세부 정보가 포함된 설명과 권장하는 코드 변경 사항이 있답니다. 또한 이 보기 세션 버튼을 클릭하고 변경 사항을 스크롤하면서 GitHub Copilot이 이 변경 사항을 구현한 전체 세션 로그도 볼 수 있어요. 실제 코드를 살펴봅시다. 음, 좋아 보여요. 하지만 피드백을 추가하고 싶다면 이슈나 풀 리퀘스트에 바로 댓글을 달 수 있으며, GitHub Copilot이 이를 살펴보고 작업에 착수할 거예요. 여기 작은 이모지를 볼 수 있죠.

동시에 두 가지 이슈를 작업하는 것이 정말 기분 좋아요. 이제 에디터로 돌아가서 이슈 번호 1의 진행 상황을 살펴보죠. 보시다시피, Copilot 에이전트 모드는 제안된 변경 사항을 검토하고, 변경할 관련 파일을 찾고, 적절하게 변경하며, 제가 원했던 스타일과 코딩 표준까지 준수할 수 있었어요. 앱으로 돌아가면 상사가 원했던 UI 변경 사항을 구현할 수 있었답니다. 작동하는지 볼게요. "유지"라고 했어요. 여러분, 보시다시피 이것은 라이브예요. 디버깅할 시간이 없지만, 이 이슈를 클릭하면 구현되었을 거라고 확신해요. Copilot이 앱에 모달을 추가할 수 있었고, 몇 분 안에 두 가지 이슈를 작업했어요. GitHub Copilot과 Azure AI Foundry에 감사해요.

그리고 사실, 여기 있는 동안 사티아의 작업을 확인할 기회는 흔치 않으니, 그가 아까 할당한 이슈를 살펴보죠. 사이트에 그룹 크기 필터를 추가하는 것을 기억하시나요? 살펴보죠. 아, 이미 여기 있네요. 보시다시피 이것이 이슈이고, Copilot이 이러한 변경 사항을 구현하기 위해 풀 리퀘스트를 열 수 있었어요. PR을 살펴보죠. 보시다시피, Copilot은 그룹 크기 필터 속성을 추가할 수 있었어요. Redis 캐싱 로직을 추가하고, 지침과 함께 일부 테스트도 추가했답니다. 정말 멋진 일이에요. 또한 앱을 스테이징 환경에 배포했으므로 배포를 살펴보죠. 행운을 빌어요.

아래로 끝까지 스크롤하면 그룹 크기 필터가 추가된 것을 볼 수 있어요. 몇 가지 수정하고 싶은 부분이 있지만, 백스테이지에서 작업한 다음 나중에 프로덕션에 푸시해서 여러분이 볼 수 있도록 할게요. 몇 분 안에 세 가지 이슈를 해결했어요. 사티아에게 다시 넘길게요. 정말 감사해요. 병렬로 열심히 일하는 것에 대해 이야기해 봅시다.

AI 시대의 웹, 'Agentic Web'이란 무엇일까요? 🌐

케빈 스콧(Kevin Scott) CTO를 모시고 이 생태계와 더욱 에이전트 기반 웹(agentic web)을 구축할 우리의 공동 기회에 대해 더 넓은 관점을 제공받는 것이 좋을 것 같았어요. 케빈, 환영합니다.

안녕하세요, 여러분을 보니 정말 기쁘네요. CTO로서 벌써 아홉 번째 빌드 참석인데, 정말 긴 시간처럼 느껴져요. 일 년에 한 번씩 이곳에 와서 여러분과 이야기하며 내성적인 성격을 잊게 되는데, 지난 한 해 동안 기술 분야에서 정말 많은 흥미로운 일들이 있었고, 앞으로도 많은 일들이 일어날 예정이라 그 열정을 여러분과 공유하고 싶어요.

![](https://mblogthumb-phinf.pstatic.net/MjAyNTA1MjBfNzcg/MDAxNzQ3NzIzNzY4MTky.CXn3cURgG4Pz7bPY7Dk8ZYodfTxhF1gV694hy0WlllAg.N6I4kBkQQ0O771XmWxBJnTAD1ihfO18JjCxy6aOjHiYg.JPEG/1000000169.jpg?type=w800)

지금 저를 정말 흥분시키는 것은 마이크로소프트가 원해서만 일어나는 것이 아니라, 우리가 지난 몇 년간 함께 작업해 온 구성 요소, 프로토콜 및 서비스가 우리가 에이전트 기반 웹이라고 부르는 것을 형성하고 있기 때문이에요. 그리고 이것이 정말 고무적인 이유는 개방적인 방식으로 일어나고 있다는 점이에요. 우리가 나아가고 있는 방향을 고려할 때, 이것은 우리 모두가 매우 원해야 하는 일이라고 생각해요. 잠시 후에 그것이 왜 중요한지 좀 더 설명할게요.

하지만 이 개방적인 에이전트 기반 웹의 아키텍처에 들어가기 전에 에이전트가 무엇인지 이야기해 봅시다. 여러분 모두 머릿속에 다양한 정의를 가지고 있을 거예요. 마치 로르샤흐 검사 같죠. 하지만 제가 항상 다시 돌아오는 것, 그리고 마이크로소프트에서 우리가 자체적으로 구축하는 에이전트와 우리 자신과 여러분 모두가 자체 에이전트를 만들도록 돕기 위해 만들려고 노력하는 기능 측면에서 우리가 추진하고 있는 것은, 에이전트란 인간이 작업을 위임할 수 있는 것이라는 점이에요. 시간이 지남에 따라 이러한 에이전트에게 위임하는 작업은 점점 더 복잡해지고 있으며, 그것이 우리가 시간이 지남에 따라 목표로 하는 것이죠.

지난 한 해 동안 우리가 본 것은 에이전트 자체의 폭발적인 증가였어요. 이것은 에이전트 기반 웹의 모습을 나타내는 우리의 다이어그램이며, 스택의 가장 위에는 우리가 구축하고 있는, 우리 파트너가 구축하고 있는, 그리고 여러분 모두가 구축하고 있는 더욱 풍부한 에이전트 생태계가 있어요. 그리고 우리는 내년에 더 많은 에이전트가 나올 것을 볼 거예요. 지난 한 해 동안 이 에이전트들에게서 일어난 놀라운 점은 그들이 이전보다 더 자주 사용되고 있다는 점이에요. 우리가 가시성을 확보하고 있는 에이전트들 전체에서 일일 활성 사용자 수는 제가 지난 빌드에서 여러분과 이야기한 이후 두 배 이상 증가했습니다. 사람들이 엄청나게 증가하고 가속화되는 속도로 사용하고 있다는 것이죠.

하지만 더 큰 변화는 지난 한 해 동안 출시된 새로운 추론 모델 덕분에 이 에이전트들이 매우 복잡한 작업을 실제로 수행할 수 있게 되었다는 점이에요. 사티아와 데모를 제공한 무대 위의 사람들에게서 이러한 내용들을 많이 보셨을 거예요. 특히 소프트웨어 엔지니어링 분야에서 이 에이전트들이 지금 할 수 있는 놀라운 일들을 말이죠. 그리고 여러분들은 빌드 기간 동안 더 많은 것을 보게 될 거예요.

하지만 이 에이전트 기반 웹 스택의 에이전트 계층 위에 **런타임 계층(runtime layer)**이라는 것이 있습니다. 런타임은 우리가 구축하고 있는 또 다른 새로운 구성 요소 집합이에요. 지난 한 해 동안 정말 좋아졌다고 방금 언급한 것은 추론 계층(reasoning layer) 또는 추론 구성 요소입니다. 여기서 런타임 계층 내에서 이러한 모델의 추론 기능은 계속해서 더 나아질 거예요.

지금 당장, 저는 추론 능력에 약간의 능력 과잉이 있다고 생각합니다. 즉, 모델이 우리가 현재 집합적으로 사용하는 것보다 더 강력합니다. 그리고 올해 빌드에서 여러분 모두에게 드리는 큰 도전 중 하나는 여러분의 야심 수준을 11로 설정하여 현재는 거의 불가능하다고 생각하는 추론 능력이 필요한 것들을 목표로 삼아 보는 것입니다. 왜냐하면 모델은 앞으로 12개월 동안 훨씬 더 강력해지고 훨씬 저렴해질 것이기 때문입니다. 따라서 우리는 이미 존재하는 능력 과잉이 더 커지지 않도록 하고 싶습니다. 하지만 추론 외에도 구축해야 할 기본적인 소프트웨어 엔지니어링 작업이 많이 있습니다.

에이전트에서 한동안 눈에 띄게 누락되었던 것은 정말 강력한 **에이전트 메모리(agentic memory)**입니다. RAG 및 긴 컨텍스트 창과 같은 방식으로 여러 번 근사화했지만, 에이전트가 실제로 수행하기를 원하는 것은 광범위한 기억력을 가지고 회상할 수 있으며, 회상에서 높은 정확성을 가질 수 있는 것입니다. 따라서 메모리에서 가져오는 것이 정확하고 신뢰할 수 있습니다. 그래서 정말 흥미로운 일들이 일어나고 있습니다. 예를 들어, 우리는 빌드에서 나중에 더 자세히 알아볼 수 있는 TypeAgent라는 시스템을 오픈 소스화했습니다. 이는 이 에이전트 메모리를 해결하려는 한 가지 방법입니다. 이는 에이전트를 덜 거래적으로 만들어 문제를 한 번 해결하고 해결책을 기억할 수 있도록 하는 데 정말 중요합니다. 에이전트는 사용자와의 상호 작용을 기억하여 개인화되고 여러분이 작업을 위임하거나 협업하는 사람에게 기대하는 것과 같은 방식으로 여러분의 선호도를 이해할 수 있습니다.

이 전체 런타임 계층, Microsoft가 이러한 구성 요소를 제공하고 커뮤니티에 참여하는 방식을 생각하면, 이러한 것들은 Azure Foundry에서 제공될 것입니다. 따라서 Azure Foundry는 앞으로 더욱 풍부해질 것이며, 제 동료 Jay Parikh와 다른 많은 사람들이 이 Foundry 계층과 이 런타임에 대해 훨씬 더 자세히 이야기하는 것을 들을 수 있을 것입니다.

하지만 개방적인 에이전트 기반 웹이 될 수 있는 것을 생각한다면 정말 중요한 것은 에이전트가 여러분을 대신하여 작업을 수행할 수 있어야 한다는 것입니다. 그리고 에이전트가 여러분을 대신하여 작업을 수행할 수 있다는 것의 정말 중요한 점 중 하나는 그들이 더 넓은 세상과 연결되어야 한다는 것입니다. 따라서 프로토콜, 즉 MCP와 A2A, 그리고 앞으로 나올 가능성이 있는 것들이 필요합니다. 이러한 것들은 여러분이 작성하는 에이전트와 지금 수억 명의 사람들이 매우 활발하게 사용하는 에이전트를 개방적이고 신뢰할 수 있으며 상호 운용 가능한 방식으로 연결하여 콘텐츠에 액세스하고, 서비스에 액세스하고, 사용자를 대신하여 작업을 수행하며, 그들에게 위임된 작업을 수행하는 데 도움이 될 것입니다.

MCP와 NLWeb, Agentic Web을 위한 핵심 기술일까요? 🔑

그래서 이번 빌드에서 우리가 정말 이야기하고 싶은 흥미로운 점은 MCP(Model Context Protocol)에 대한 우리의 진지한 노력입니다. MCP는 지난 몇 달 동안 엄청나게 성장했으며, 그럴만한 이유가 있습니다. 이 개방적인 에이전트 기반 웹 생태계에서 매우 중요한 틈새를 채우고 있습니다. 인터넷의 HTTP와 유사하게 매우 간단한 프로토콜로, 프로토콜 자체에 운반하는 페이로드에 대한 의견이 많지 않기 때문에 정말 정교한 작업을 수행할 수 있습니다. 이는 또한 그 위에 레이어링할 수 있는 훌륭한 기반이 된다는 것을 의미하며, 인터넷 자체가 어떻게 나타났는지 생각하면 구성 요소의 구성 가능성과 프로토콜의 레이어링이 풍부함에 도달하는 데 정말 중요했습니다.

이는 좋은 시작점을 제공하고, 시간이 지남에 따라 개발될 때 발견하는 문제를 해결할 수 있는 능력을 제공합니다. 따라서 사티아로부터 Windows의 MCP 레지스트리에 대해 방금 들었습니다. 우리는 Microsoft의 퍼스트 파티 플랫폼 구성 요소 및 퍼스트 파티 서비스가 MCP를 지원하도록 많은 노력을 기울이고 있습니다. 우리는 에이전트 간 및 에이전트와 서비스 간의 에이전트 기반 통신을 위한 표준 프로토콜로 선택했으며, 앞으로 몇 달 동안 Anthropic의 파트너 및 협력자들과 함께 MCP와 같은 프로토콜 위에서 해결해야 하는 정말 어려운 엔터프라이즈 문제를 해결할 수 있도록 많은 노력을 기울일 것입니다. 예를 들어, 에이전트가 자신을 식별하는 방법, 에이전트가 액세스해야 하는 시스템에 액세스할 수 있도록 정말 풍부한 권한 부여 인프라를 구축하는 방법 등입니다. 그리고 우리는 여기에서 개방형 커뮤니티를 정말 지원할 것입니다. 부분적으로는 스택의 이 계층에서 프로토콜의 유용성을 생각할 때 가장 중요한 것은 **보편성(ubiquity)**이기 때문입니다.

우리는 모든 종류의 논쟁에 참여할 수 있으며, 엔지니어로서 여러분 모두 이것을 완벽하게 할 수 있다는 것을 알고 있으며, 저도 마찬가지입니다. 여러분은 기술 조각에 대해 정말 날카로운 의견을 가지고 있고, 이것이 저것보다 조금 더 낫다는 것을 알고 있습니다. 하지만 이 모든 것보다 더 나은 것은 우리가 모두 사용하고 그 위에 구축할 수 있는 표준을 확보하는 것입니다. 그리고 우리는 그것이 MCP이기를 바랍니다.

다음으로 이야기하고 싶은 것은 레이어링에 대해 언급했다는 것입니다. 웹으로 돌아가서 생각해보면 HTTP가 있고, 그 위에 HTTP 위에 있는 것들, 주로 HTML과 같은 것들이 있습니다. 이것은 페이로드에 대해 의견을 가지고 있습니다. 그리고 오늘 우리는 NLWeb을 발표하고 있으며, GitHub 저장소에서 코드를 확인해보세요. NLWeb의 아이디어는 웹사이트 또는 API를 이미 가지고 있는 모든 사람이 웹사이트 또는 API를 매우 쉽게 에이전트 기반 애플리케이션으로 만들 수 있도록 하는 것입니다.

이것은 대규모 언어 모델의 모든 기능을 구현하고 활용하여 이미 제공하고 있는 서비스와 제품을 풍부하게 할 수 있도록 해줍니다. 그리고 모든 NLWeb 엔드포인트는 기본적으로 MCP 서버이기 때문에 사람들이 NLWeb을 통해 제공하는 것들이 MCP를 사용하는 모든 에이전트에게 접근 가능하다는 것을 의미합니다. 따라서 이것을 에이전트 기반 웹을 위한 HTML처럼 생각할 수 있습니다.

우리는 이미 NLWeb을 사용하여 빠른 구현 및 프로토타입을 매우 빠르게 수행할 수 있었던 파트너들과 많은 작업을 수행했습니다. TripAdvisor, O'Reilly Media 등 인터넷에서 중요한 제품과 서비스를 제공하는 많은 훌륭한 회사들과 협력하여 사이트에 NLWeb 기능을 추가하여 사이트에서 직접 에이전트 경험을 할 수 있도록 했습니다.

여러분 모두에게 역량을 보여주기 위해 제가 보여주고 싶었던 것 중 하나는 지금 당장, 사티아의 기조 연설의 놀라운 마무리를 들을 때까지 기다려 주세요. 그 직후에 이 저장소에서 코드를 가져올 수 있습니다. 그리고 여기서 보여주는 것은 액세스할 수 있는 모든 GitHub 저장소 위에 에이전트 기반 NLWeb 사용자 인터페이스를 배치하는 방법입니다. 화면에 보여주는 것은 이 Python 스크립트를 실행하면 GitHub API와 통신하고 GitHub 저장소에서 많은 데이터를 추출합니다. 여기서 CTO 사무실 팀원인 Jennifer Marsman이 저장소의 모든 메타데이터와 데이터를 NLWeb 저장소로 가져오는 것을 볼 수 있습니다. 그녀는 지구상 최고의 데모 제작자입니다.

두 번째로 하는 일은 이 모든 데이터를 가져와 이 경우에 임베딩을 계산하는 작은 스크립트를 실행하는 것입니다. 여기서 어떤 모델을 사용하고, 어떤 임베딩을 사용하고, NLWeb 서버가 액세스할 수 있도록 임베딩을 저장하는 벡터 스토어에 대해 많은 선택권이 있습니다. 임베딩을 계산하고 이 경우 로컬 벡터 스토어에 가져오면 노트북에서 실행되는 것입니다. 그런 다음 그 위에 몇 가지 흥미로운 쿼리를 만들 수 있습니다. 이 경우 데이터베이스 쿼리 또는 어떤 종류의 정보 검색만으로는 얻을 수 있는 것보다 더 많은 것을 할 수 있습니다. 예를 들어, "제가 이 코드를 작성했습니다. 어떤 PR에 있었습니다. 기억이 나지 않습니다. 이 알고리즘 이름을 기억해야 합니다"라고 말할 수 있으며, 그것이 NLWeb이 할 수 있는 종류의 것입니다.

그리고 우리가 솔직히 상상조차 할 수 없는 것들이 있습니다. 이것이 바로 우리가 이것을 오픈 소스화하고 있으며, NLWeb을 개선하고 더 나은 것으로 만들 수 있는 방법에 대한 여러분 모두의 많은 피드백을 듣기를 바라는 이유입니다. 다시 말하지만, 이 스택의 이 계층에서 프로토콜의 유용성을 생각할 때 가장 중요한 것은 보편성이기 때문입니다. 우리는 모든 종류의 논쟁에 참여할 수 있으며, 엔지니어로서 여러분 모두 이것을 완벽하게 할 수 있다는 것을 알고 있으며, 저도 마찬가지입니다. 여러분은 기술 조각에 대해 정말 날카로운 의견을 가지고 있고, 이것이 저것보다 조금 더 낫다는 것을 알고 있습니다. 하지만 이 모든 것보다 더 나은 것은 우리가 모두 사용하고 그 위에 구축할 수 있는 표준을 확보하는 것입니다. 그리고 우리는 그것이 MCP이기를 바랍니다.

사티아에게 넘기기 전에 마지막으로 하고 싶은 말은 개방성이 왜 그렇게 중요한지에 대한 이 두 가지 점을 강조하는 것입니다. 간단한 구성 요소와 서로 구성 가능한 간단한 프로토콜이 거기에 있고, 참여하고 싶은 모든 개발자 또는 아이디어가 있는 모든 개발자의 완전한 검토와 창의성에 노출될 때 세상에서 어떤 일이 일어날 수 있는지는 믿을 수 없습니다.
